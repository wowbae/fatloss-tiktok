## 🎨 ТРЕБОВАНИЯ К ДИЗАЙНУ (для лендингов и веб-приложений)

### Визуальная система
- **Шрифты:** Система из 2–3 шрифтов — display для заголовков, body для текста, mono для кода/данных. Избегать Inter, Roboto, Arial.
- **Палитра:** CSS-переменные для цветов. Доминирующий цвет + акценты. Избегать шаблонных сочетаний (фиолетовый на белом).
- **Noise overlay:** Полупрозрачный слой шума/зернистости для глубины и тактильности.

### Ключевые UI-паттерны
- **Floating morphing navbar:** Навбар с плавной морфинг-анимацией, меняет форму/прозрачность при скролле.
- **Cinematic hero:** Кинематографичный hero-блок с параллаксом, градиентами, крупной типографикой.
- **Rotating diagnostic cards:** Карточки с лёгким 3D-вращением или hover-эффектом.
- **Telemetry typewriter:** Эффект печатной машинки для данных/статусов.
- **Animated scheduler cursor:** Анимированный курсор или индикатор в планировщиках/календарях.
- **Manifesto reveal:** Поэтапное появление текста/блоков при скролле (scroll-triggered).
- **Sticky protocol archive:** Липкий блок с архивом/протоколом при скролле.
- **Pricing section:** Чёткая секция тарифов с визуальной иерархией.
- **Operational footer:** Футер с навигацией, контактами, соцсетями.

### Интерактивность
- **Magnetic button behavior:** Кнопки с магнитным эффектом — курсор притягивает элемент при наведении.
- **GSAP:** Использовать GSAP для сложных анимаций (timeline, scroll-triggered, stagger).

### Принципы
- Асимметрия и неожиданные акценты вместо центрированных карточек.
- Щедрый negative space или контролируемая плотность.
- Анимации: staggered entrance, scroll-triggered reveals, выразительные hover-состояния.

## 🏗 ЧАСТЬ 2: АРХИТЕКТУРНЫЕ ПРИНЦИПЫ (не менять)

### 1. Модульность
- Каждый модуль независим — можно удалить без поломки системы
- Нет циклических зависимостей между модулями
- Модули общаются через чёткие интерфейсы (не напрямую)

### 2. Слабая связанность
- Зависимости внедряются через параметры/контекст
- Нет прямых импортов из глубокой вложенности (>3 уровней)
- Интерфейсы важнее реализации

### 3. Единая ответственность
- **Один файл = одна задача**
- Файл >200 строк → разделить на модули
- Функция >50 строк → рефакторить

### 4. Конфигурация
- Все изменяемые значения в `shared/constants/` или `.env`
- **Магические числа запрещены** — выносить в константы
- Секреты и API-ключи — только через `.env`

### 5. Обработка ошибок
- Try-catch на границах модулей
- Логи ошибок с контекстом (id, timestamp)
- Не игнорировать ошибки (catch без логирования запрещён)

### 6. Оптимизация запросов
- **Объединять** — несколько запросов к одной сущности → один запрос с нужными полями
- **Параллелить** — независимые запросы → `Promise.all()` вместо последовательных `await`

## 📁 ЧАСТЬ 3: СТРУКТУРА ПРОЕКТА (адаптировать под проект)

### Базовый шаблон

```
project-name/
├── docs/                    # Документация
│   ├── PROJECT_PLAN.md      # Детальный план разработки
│   └── RULES.md             # Правила проекта (опционально)
├── shared/                  # Общий слой (константы, типы, утилиты)
│   ├── constants/           # Константы по категориям
│   │   ├── [feature].ts     # Константы по фичам
│   │   └── index.ts
│   ├── types/               # TypeScript типы
│   │   ├── [entity].ts
│   │   └── index.ts
│   └── utils/               # Общие утилиты
│       ├── logger.ts        # Логирование
│       └── [util].ts
├── backend/                 # Бэкенд (или server/ для простых проектов)
│   ├── [feature]/           # Группировка по фичам
│   │   └── [module].ts
│   └── init.ts              # Точка входа
├── frontend/                # Фронтенд (если есть)
│   ├── src/
│   │   ├── components/
│   │   ├── routes/
│   │   └── ...
│   └── tsconfig.json
├── prisma/
│   ├── schema.prisma
│   └── client.ts
├── .env
├── .env.example
├── package.json
└── tsconfig.json
```

### Правила структуры
- **Группировка по фичам**, не по типам файлов
- Максимальная глубина: 3 уровня
- `shared/` — всё, что используется в нескольких модулях
- `constants/` — единый источник правды для конфигурации

## 📝 ЧАСТЬ 4: СТАНДАРТЫ КОДА (не менять)

### Именование
- Понятное, на **английском языке**
- Без транслита (никаких "kartochka", "tovar")
- `camelCase` — переменные и функции
- `PascalCase` — типы, интерфейсы, классы
- `UPPER_CASE` — константы

### Комментарии
- **На русском языке** в ключевых местах
- Объясняют «почему», а не «что»
- Ключевые места: экспорты, сложные алгоритмы, обработчики ошибок

### TypeScript
- Строгий режим (`strict: true`)
- Интерфейсы вместо типов
- Const-объекты вместо enum
- Запрещено использовать `any`

### Стиль кода
- Функциональное программирование вместо ООП
- Названия переменных с глаголами (`isLoading`, `hasError`)
- Early returns для избегания вложенности
- Happy path в конце функции
